Design Notes

Key Changes from Current Design

  • Tabs replace inline links: "All Meetings", "Governing Body", "Public Works" become proper tab selectors instead of navigation links to separate pages.
  • Two-row structure: Row 1 = tabs (what you're viewing), Row 2 = controls (how you're viewing it). Clear separation of concerns.
  • "All Meetings" tab: Shows the current home page content with month picker, search, category filter, and stats.
  • Committee tabs: Inline the committee content (currently on /governing-body, /public-works) into the same page. No page navigation needed.
  • Contextual controls: When a committee tab is active, the filter dropdown is hidden (already filtered), search placeholder updates, and a Follow button appears.

Mobile Optimizations

  • Tabs scroll horizontally with hidden scrollbar and fade-edge hint.
  • All touch targets remain 44px minimum.
  • Compact month display ("Mar '26") in sticky header.
  • Three-row sticky layout: tabs, controls, search — each row has a clear purpose.
  • Color dots replace full SVG icons for committee tabs on mobile (saves space).

Scalability

  • Horizontal scroll means unlimited tabs without layout breakage.
  • Tabs are driven by the COMMITTEES config — adding a new committee automatically adds a tab.
  • The "All Meetings" tab always stays first as the default view.

Implementation Approach

  • Option A (URL-based): Tabs change URL params (?tab=all, ?tab=governing-body). Committee pages (/governing-body, /public-works) redirect to home with the tab param. Preserves shareability.
  • Option B (Client-state): Tabs are pure client state. Faster switching, no URL change. Committee pages still exist as separate routes but the tab UI provides quick switching.
  • Recommended: Option A — URL-based tabs give you shareable links, back-button support, and SSR-friendly routing.
